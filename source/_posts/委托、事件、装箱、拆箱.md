## 一、值类型和引用类型

值类型包括所有整数、浮点数、`bool`，以及用 `struct` 声明的结构。这里要特别留意 `struct`——它是经常容易犯错误的地方。很多人会把 `struct` 当作类来使用，这是完全错误的。因为它是值类型，在赋值操作时通过**直接拷贝数据**完成，所以常会出现这样的困惑：`a`、`b` 同是结构体实例，`a` 赋值给 `b`，当 `b` 更改了数据后却发现 `a` 的数据没有同步。事实上它们根本就是两块独立的数据空间——当 `a` 赋值给 `b` 时并不是引用拷贝，而是**整个数据空间的拷贝**，相当于有了两个不同的西瓜，只是长得比较像而已。

```csharp
// struct 是值类型，赋值 = 完整拷贝
struct Point
{
    public int X;
    public int Y;
}

Point a = new Point { X = 1, Y = 2 };
Point b = a;          // 把 a 的所有数据完整拷贝给 b
b.X = 10;             // 修改 b 的 X

Console.WriteLine(a.X);  // 输出 1 —— a 不受影响
Console.WriteLine(b.X);  // 输出 10
```

引用类型包括类、接口、委托（委托也是类）、数组，以及内置的 `object` 与 `string`。在 C# 里万物皆是类——虽然 `int` 等值类型本质上也是类，只不过它们是特殊的、值类型的类。"类都是引用类型"虽然不算完全精确，但也不妨碍它作为一个好记的口诀。

值类型的变量会**直接存储数据**，而引用类型的变量持有的是**数据的引用**，其真实数据存储在托管堆中。当声明一个引用类型变量时，只在栈（或堆）中分配一小片内存用于容纳一个地址，此时并没有为其分配堆上的内存空间，因此它为 `null`；直到使用 `new` 创建实例时，才在堆上分配空间，并把堆上空间的地址保存给这个引用变量，这时引用变量才真正指向了有效的内存。

这也解释了为什么 `struct` 和类的赋值表现完全不同。`struct` 赋值时，是把整块数据拷贝过去——被赋值的变量已经在一块全新的内存里存储了数据，与原变量再无瓜葛。而类的赋值，只是把数据的**引用**拷贝过去——被赋值的变量和赋给它的变量里存着**同一个地址**，指向堆上的同一块数据空间。所以 `b` 更改了数据之后，再用 `a` 访问，看到的也是被改过的那块数据。

## 二、委托和事件

### 委托

在 C# 中万物皆是类，大部分时间里都看不到指针的身影，最多也只是引用，因为指针被封装在内部函数当中。但回调函数的需求依然存在，于是 C# 引入了**委托（delegate）**的概念，所有函数指针的功能都以委托的方式来达成。委托可以被视为一个更高级的函数指针——它不仅能将地址指向另一个函数，还能传递参数、获取返回值等。系统还为委托对象自动生成了同步和异步的调用方式，开发人员使用 `BeginInvoke`、`EndInvoke` 方法就可以避开 `Thread` 类，直接实现多线程调用。

#### 委托的常见语法

```csharp
// 1. 声明委托类型
public delegate int CalculateDelegate(int x, int y);

// 2. 创建委托实例
CalculateDelegate calc = new CalculateDelegate(Add);

// 3. 使用委托——像调用方法一样
int result = calc(3, 5);        // 输出 8
result = calc.Invoke(3, 5);     // 与上一行等价

// 4. 多播委托 —— 一个委托挂载多个方法，Invoke()会按顺序依次调用委托中的所有方法
calc += Subtract;               // 添加方法
calc -= Add;                    // 移除方法

static int Add(int a, int b) => a + b; // lambda 表达式
static int Subtract(int a, int b) => a - b;
```

不过在实际开发中，较少直接使用自定义委托，而是使用 C# 为我们准备好的 `Action` 和 `Func` 这两个泛型委托。两者的区别在于：

| 委托 | 返回值 | 示例 |
|------|--------|------|
| `Action` | 无返回值 | `Action<string> print = Console.WriteLine;` |
| `Action<T1, T2, ...>` | 无返回值 | `Action<int, int> log = (x, y) => Console.WriteLine(x + y);` |
| `Func<T, TResult>` | 有返回值（最后一个泛型参数是返回类型） | `Func<int, int, int> add = (a, b) => a + b;` |

```csharp
// Action 示例 —— 无返回值
Action<string> log = msg => Console.WriteLine($"[LOG] {msg}");
log("Hello");   // 输出：[LOG] Hello

// Func 示例 —— 有返回值
Func<int, int, int> multiply = (a, b) => a * b;
int product = multiply(3, 4);   // 12
```

#### 委托的底层实现

我们在创建委托时，实际上就是创建了一个委托类实例。这个委托类继承自 `System.MulticastDelegate`，类实例中包含 `BeginInvoke`、`EndInvoke`、`Invoke` 三个方法，分别对应异步开始调用、结束异步调用和直接调用。

`Delegate` 类中有一个变量用于存储函数地址。当使用 `=` 操作时，把函数地址赋值给该变量存起来。不过这个存储函数地址的变量实际上是一个**调用列表（invocation list）**，其内部结构为一个链表：委托实例持有 `_prev` 字段指向链表中的前一个委托节点。当链表中只有一个方法时，`_prev` 为 `null`；当再 `+=` 一个方法时，实际是创建了一个新的 `MulticastDelegate` 实例，其 `_prev` 指向原来的实例，以此串成链表。注意，每 `+=` 一次都会生成一个新实例并串到链表头部，所以 `_prev` 链的方向是**新 → 旧**。当委托被调用（`Invoke`）时，CLR 会先沿着 `_prev` 链走到最末端（即最早添加的节点），然后**从旧到新**依次调用每个节点的方法，保证回调顺序与注册顺序一致。

`MulticastDelegate` 还重写了 `+=` 和 `-=` 这两个操作符，其内部实现对应的就是 `Delegate.Combine` 和 `Delegate.Remove` 静态方法：对委托执行 `+=` 相当于把新的函数地址加入调用列表；执行 `-=` 则是从调用列表中移除该函数。

看到这里我们就彻底明白了：`delegate` 关键字本质上是一个语法糖，背后由 C# 编译器将其重写为一个继承自 `System.MulticastDelegate` 的类。

### 事件

#### 事件的常见语法

```csharp
// 1. 声明事件（基于已有的委托类型）
public event Action<string> OnMessageReceived;

// 2. 注册事件
OnMessageReceived += HandleMessage;

// 3. 注销事件
OnMessageReceived -= HandleMessage;

// 4. 触发事件（只能在声明事件的类内部调用）
OnMessageReceived?.Invoke("Hello World");

// 事件处理方法
void HandleMessage(string msg)
{
    Console.WriteLine($"收到消息：{msg}");
}
```

#### 事件的底层实现

`event` 很简单，它在委托（delegate）的基础上又做了一次封装。这次封装的意义在于：**限制用户直接操作委托实例中变量的权限**。

如果不使用 `event`，委托字段就像一个普通的公共变量，外部代码可以直接对它执行 `=` 赋值，从而清空之前所有订阅的方法。而 `event` 关键字本质上会生成类似属性（property）的 `add` 和 `remove` 访问器，将底层的委托字段设为私有，外部只能通过 `+=`/`-=` 来增减方法，无法直接赋值覆盖。其原理大致如下：

```csharp
// event 背后的等价封装（简化示意）
private Action<string> _onMessage;   // 底层委托字段是私有的

public event Action<string> OnMessage
{
    add    { _onMessage += value; }   // 只能通过 += 注册
    remove { _onMessage -= value; }   // 只能通过 -= 注销
}
```

封装之后，用户不再能够用赋值操作（即 `=` 操作符）来直接改变委托变量，只能通过注册或注销委托的方式来增减委托函数的数量。也就是说，被 `event` 声明的委托不再对外暴露 `=` 操作，但仍然保留了 `+=` 和 `-=` 操作符可供使用。

为什么要做这样的限制呢？在平时的编程中，项目庞大、经手人员众多，我们常常无法得知其他人在编写的代码意图是什么。如果委托字段直接暴露在外，随时可能被 `=` 赋值而**清空前面累积起来的委托链表**，操作的权限范围太大，导致 bug 难以排查。而声明为 `event` 后，编译器在内部重新封装了委托，让暴露在外的委托不再有被随时清空和重置的风险——只能一个个注册、一个个注销，这样就保证了"谁注册谁负责销毁"的秩序，更好地维护了委托链的完整。

## 三、装箱和拆箱

什么是装箱和拆箱？其实很简单：**把值类型实例转换为引用类型实例，就是装箱；反过来，把引用类型实例转换为值类型实例，就是拆箱。**

举个例子：

```csharp
int a = 5;
object obj = a;     // 装箱
```

`a` 是值类型，是直接持有数据的变量；`obj` 是引用类型，指针与数据是分离的。把 `a` 赋值给 `obj`，实际上就是 `obj` 在堆上开辟了一块新的数据空间，拷贝了 `a` 的数据，并为自己创建了一个指针指向这块数据空间。接着上面的代码继续：

```csharp
int b = (int)obj;   // 拆箱
```

这就是拆箱——把 `obj` 指向的内存空间复制一份交给 `b`。因为 `b` 是值类型，它不允许指向某个内存空间，只能靠复制数据来传递数据。

### 堆内存和栈内存

这里稍微阐述一下堆和栈内存，因为很多人对此有错误的认识。为什么要有栈内存和堆内存之分？用简短的语言来说明：

> 栈是本着**先进后出（LIFO）**原则的存储机制，它是一段**连续的内存**，所以对栈数据的定位比较快速。而堆则是**随机分配**的空间，处理的数据比较多，无论如何至少要两次定位。堆内存的创建和删除节点的时间复杂度是 O(log n)，栈创建和删除的时间复杂度则是 O(1)，栈速度更快。
>
> 那么既然栈速度这么快，全部用栈不就好了？这又涉及生命周期的问题。栈中的生命周期是必须确定的，销毁时必须按次序从最后分配的块开始销毁，创建后什么时候销毁必须是一个定值，所以在分配和销毁时不灵活，基本只用于函数调用和递归调用这类生命周期比较确定的地方。相反，堆内存可以存放生命周期不确定的内存块，满足"需要时再删除"的需求，所以堆内存更适合全局性质的、需要灵活分配和销毁的内存块。

很多人把"值类型放在栈中、引用类型放在堆中"当作绝对规律，这是错误的。栈内存主要为**确定性生命周期**的内存服务，堆内存则更多服务于**无序的、随时可能释放**的内存。因此值类型既可以在堆内也可以在栈内，引用类型的指针部分同样可以在栈内或堆内——但引用类型**指向的内存块**一定在堆内。

### 什么时候会用到装箱

大部分时候只有当程序、逻辑或接口需要更加通用时，才会需要装箱。比如：

```csharp
// 情况一：调用含 object 类型参数的方法
void PrintObject(object obj)
{
    Console.WriteLine(obj);
}

int value = 42;
PrintObject(value);     // int → object，发生装箱

// 情况二：非泛型容器，元素类型定义为 object
ArrayList list = new ArrayList();
list.Add(100);          // int → object，发生装箱
list.Add("hello");      // string 本身就是引用类型，不装箱
```

当你需要将一个值类型（如 `int`）传入参数为 `object` 的方法时，就需要装箱。又比如一个非泛型的容器（如 `ArrayList`）为了保证通用，将元素类型定义为 `object`，值类型数据放入时就会触发装箱。

### 装箱和拆箱的内部操作

**装箱**：根据相应的值类型在堆中分配一个内存块，再将数据拷贝给它。具体按以下三步进行：

1. **第一步**：在堆内存中新分配一个内存块（大小为值类型实例大小加上一个方法表指针和一个 `SyncBlockIndex`）。
2. **第二步**：将值类型的实例字段拷贝到新分配的内存块中。
3. **第三步**：返回堆中新分配对象的地址——这个地址就是一个指向对象的引用了。

**拆箱**则更为简单：先检查对象实例，确保它是给定值类型的一个装箱值，再将该值从实例复制到值类型变量的内存块中。

### 装箱、拆箱对执行效率的影响及优化

由于装箱和拆箱时生成的是全新的对象，不断地分配和销毁内存不仅大量消耗 CPU，也会增加内存碎片，从而降低性能。那该如何优化呢？

最关键的就是**减少装箱和拆箱操作**——在编程规范中要牢记：装箱和拆箱是消耗较大的操作，写代码时需要特别留意。

整数、浮点数、布尔等数值型变量变化手段不多，主要靠加强编码规范来减少装拆箱。而 `struct` 有点不一样——它既是值类型，又可以像类一样实现接口，用途更多样、转换途径也更多，稍不留神就会埋下性能隐患。所以这里重点讲讲 `struct` 的优化方法。

#### 通过重载函数来避免装箱

比如常用的 `ToString()`、`GetHashCode()`、`Equals()` 方法：如果 `struct` 没有重写这些方法，调用它们时会先装箱再调用，导致内存分配的性能损耗。

为什么 struct 调用这些方法需要先装箱？因为这些方法的默认实现定义在 `System.Object` 或 `System.ValueType` 基类中，都是虚方法（`virtual`）。当 struct 没有重写它们时，虚方法调用需要查方法表（vtable），而方法表存储在引用类型对象中——值类型实例本身不携带方法表指针。因此必须先将 struct 装箱为一个引用类型对象，才能完成虚方法的派发。例如：

```csharp
struct MyStruct
{
    public int Value;
}

MyStruct s = new MyStruct { Value = 10 };
s.ToString();     // MyStruct 没有重写 ToString → 先装箱再调用 → 产生 GC 垃圾
```

而要避免这一点很简单——在 struct 中手动重写这些方法即可：

```csharp
struct MyStruct
{
    public int Value;

    public override string ToString() => Value.ToString();  // 重载后不再装箱
    public override int GetHashCode() => Value.GetHashCode();
    public override bool Equals(object obj) => obj is MyStruct s && Value == s.Value;
}
```

所以，对于那些需要调用的引用类型方法，struct 必须逐一**手动重载**。

#### 通过泛型来避免装箱

别忘了 struct 也是可以实现接口的。在不同的、相似的、具有父子关系的 struct 之间，可以用泛型来传递参数，这样就不用装箱后再传递了。

```csharp
interface IPrintable { void Print(); }

struct Document : IPrintable
{
    public string Name;
    public void Print() => Console.WriteLine($"打印：{Name}");
}

// 不使用泛型 —— 传参时装箱
void ProcessByInterface(IPrintable printable)
{
    printable.Print();
}

// 使用泛型 —— 不会装箱
void ProcessByGeneric<T>(T printable) where T : IPrintable
{
    printable.Print();
}

Document doc = new Document { Name = "报告" };
ProcessByInterface(doc);   // doc 转为 IPrintable → 装箱
ProcessByGeneric(doc);     // 泛型直接约束为 IPrintable，不装箱
```

为什么传给 `IPrintable` 类型的参数就需要装箱？原因是 struct 实例本质上只是栈上的一段**纯数据**，没有对象头（Object Header），也没有方法表指针（MethodTable Pointer）。而接口方法调用属于**虚分派（virtual dispatch）**，CLR 必须通过方法表来查找实际要调用的方法——栈上的 struct 不携带这些信息，所以必须先装箱，在堆上生成一个带有完整对象头和接口映射表（Interface Map）的包装对象，才能完成接口分派。

```csharp
// 通过具体类型直接调用 —— 不装箱
Document doc = new Document();
doc.Print();     // 编译器知道 doc 就是 Document，直接静态调用

// 通过接口引用调用 —— 需要装箱
IPrintable ip = doc;   // doc 装箱：在堆上创建带方法表指针的副本
ip.Print();            // 通过接口分派，走方法表查找
```

而泛型则绕过了这个问题。由于 JIT 编译时会对每个值类型生成**专用版本**（代码特化），当 `T` 是 `Document` 时，JIT 生成的方法内部等价于直接接收 `Document` 类型的参数。调用 `Print()` 时走的是**具体类型的静态分派**，而不是接口虚分派——`where T : IPrintable` 约束仅用于编译期检查，确保 `T` 实现了 `Print()`，运行时完全不经过接口查表，因此全程不需要装箱。

#### 通过继承统一的接口提前装箱，避免多次重复拆装箱

很多时候拆装箱不可避免，那么我们就让多种 struct 继承某个统一的接口。这样不同 struct 就可以用相同的接口访问。将 struct 传递到其他方法时，相当于**提前进行了一次装箱操作**——在方法中得到的是引用类型的值，并且具备所需的接口，避免了在方法内部重复多次的拆装箱。

```csharp
interface IShape
{
    double Area();
}

struct Square : IShape
{
    public double Side;
    public double Area() => Side * Side;
}

struct Circle : IShape
{
    public double Radius;
    public double Area() => Math.PI * Radius * Radius;
}

// Test 方法接收 IShape 引用类型，struct 传入时装箱一次
void Test(IShape shape)
{
    // 方法内部可以多次调用 shape.Area()、shape 的方法等
    // 不会再发生装箱，因为 shape 已经是引用类型
    Console.WriteLine($"面积：{shape.Area()}");
    Console.WriteLine($"类型：{shape.GetType().Name}");
}

Square square = new Square { Side = 5 };
Circle circle = new Circle { Radius = 3 };

Test(square);   // Square → IShape，装箱一次，内部不再重复装箱
Test(circle);   // Circle → IShape，装箱一次，内部不再重复装箱
```

对比一下：如果没有统一的接口，Square 和 Circle 又都需要传入同一个方法处理，就只能退回到用 `object` 来接收：

```csharp
// 没有统一接口 → 用 object 凑合
void ProcessWithoutInterface(object obj)
{
    // 每次访问具体字段/方法，都要先拆箱
    if (obj is Square)
    {
        Square s = (Square)obj;         // 拆箱！
        Console.WriteLine($"正方形面积：{s.Side * s.Side}");
    }
    else if (obj is Circle)
    {
        Circle c = (Circle)obj;         // 拆箱！
        Console.WriteLine($"圆形面积：{Math.PI * c.Radius * c.Radius}");
    }
    // 如果方法里还要再做其他操作（比如打印类型名、序列化等），
    // 每次都从 object 里拆出来，拆箱次数 = if 分支数 × 操作次数
}

Square square = new Square { Side = 5 };
Circle circle = new Circle { Radius = 3 };

ProcessWithoutInterface(square);  // square → object（装箱）
ProcessWithoutInterface(circle);  // circle → object（装箱）
// 更糟的是：方法内部每次 (Square)obj / (Circle)obj 都在拆箱
// 如果用 is 做类型判断再强转，等于拆了两次（is 一次，强转又一次）
```

可以看到，没有统一接口时：传入方法时**装箱一次**，但方法内部每次要访问具体数据就得**拆箱一次**——如果有 N 种 struct 类型、每种有 M 个操作，最坏情况下拆箱次数 = N × M。而使用统一接口后，只需在入口处装箱一次，整个方法体内都在跟已装箱的引用类型打交道，零额外开销。

这种做法的优势在于：装箱发生在**方法入口处**，只做一次。方法内部操作的都是已装箱好的引用类型对象，无需再次拆装箱，从而将性能影响控制在最小范围。

<div class="tip">
最后依然要提醒大家：struct 值类型数据结构如果没有真正理解它的原理，用起来可能会引发很多麻烦。切记不要盲目认为使用结构体一定会让性能提升——在没有完全彻底理解之前贸然大量使用 struct，可能会对你的程序性能带来重创。
</div>
