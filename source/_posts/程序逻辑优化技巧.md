## 一、使用 List 和 Dictionary

`List` 和 `Dictionary` 是 C# 中最常用的集合类型，它们的底层都是由数组实现的。理解它们的内部机制，才能避免在性能敏感的场景中踩坑。

### List

`List` 在插入元素时，需要将插入位置之后的所有元素依次向后移动，因此 `Insert` 的时间复杂度是 **O(n)**。

`Contains` 通过遍历数组来查找目标元素，时间复杂度也是 **O(n)**。

`Remove` 同样需要先找到元素再移动后续元素，复杂度同样是 **O(n)**。

> 因此，如果频繁调用 `Insert`、`Contains`、`Remove` 这些 O(n) 操作，尤其是在 List 容量较大的情况下，会带来很多不必要的性能消耗。在合适的场景下，可以考虑使用 `HashSet`、`Dictionary` 等 O(1) 查找的数据结构来替代。

### Dictionary

在编写程序时应注意给 `Dictionary` 设置一个合理的初始容量，而不是任由它自行扩容。扩容时的数组重新分配不仅会加重 **GC** 的负担，过小的容量还会让哈希冲突变得更加频繁，进一步导致性能损耗。

此外，如果使用**类实例**作为 `Dictionary` 的 Key，`Dictionary` 会使用 `Object` 的 `GetHashCode()` 方法来获取类实例的哈希值。而 `GetHashCode()` 是通过算法将内存地址转化为哈希值的，它并不会缓存计算结果——每次调用都会重新计算一次，这是一种比较隐蔽的性能损耗。如果频繁通过 `GetHashCode` 来提取 Value，我们应该关注一下 `GetHashCode` 的计算开销，考虑是否可以用唯一 ID（如 `int` 或 `string`）的方式来代替 `GetHashCode` 算法。

## 二、使用 Struct

`Struct` 是值类型。当作为局部变量使用时，`Struct` 分配的内存在**栈**上，生命周期结束后回收也十分简单——只需将栈的尾指针回退即可。与分配在堆上的类实例不同，值类型的分配和回收不会产生内存碎片，也不需要 GC 介入。

另外，CPU 从栈上读取数据比从堆上读取数据更高效，因为栈内存具有更好的访问局部性。

### CPU 缓存命中和 Struct 数组

CPU 在读取内存时，会把连续的一大块内容加载到缓存中。从缓存读取数据比直接从内存读取快得多（可达 100 倍以上）。下次读取数据时，CPU 会先在缓存中查找，如果命中则无需再次访问内存。

> 连续内存在 CPU 读取时，缓存的命中率高；而非连续内存则命中率较低。我们应该尽可能让 CPU 读取数据时命中缓存，从而提升访问效率。

正因为如此，`Struct` 数组对提高内存访问速度很有帮助。由于 `Struct` 是值类型，`Struct` 数组中的所有元素都是连续存储的——每个元素的数据实实在在"嵌"在数组的连续内存空间中。而 Class 数组则不同：虽然数组中的引用（指针）变量是连续的，但每个引用所指向的、位于堆上的实际数据则像碎片一样分散在各处。

### 使用值类型数组

不过，并非所有 `Struct` 都能提高缓存命中率。如果一个 `Struct` 太大，超过了 CPU 一次缓存加载的数据块大小，那么缓存的效果就会大打折扣——因为一次加载的数据可能只包含一个甚至半个 Struct。因此，很多高性能架构彻底抛弃了在对象上直接存放数据的方式，转而使用**值类型连续数组**来提高 CPU 缓存命中率：将同一类数据集中到基本类型的数组中，而具体对象上只存放一个索引值，通过索引来间接访问数组中的数据。我们来看一个例子：

```csharp
// 传统 OOP 方式：每个对象持有自己的数据
class A
{
    public int a;
    public float b;
    public bool c;
}

// 改进方式：对象只持有索引，实际数据集中存储在数组中
class B
{
    public int index;
}

// 数据中心：将所有同类型数据以值类型数组的形式集中存放
class C
{
    private static C _instance;
    public static C instance
    {
        get
        {
            if (null == _instance)
            {
                _instance = new C();
            }
            return _instance;
        }
    }

    // 所有数据集中存放在值类型数组中，内存连续
    public int[] a = new int[] { 2, 3, 5, 6 };
    public float[] b = new float[] { 2.1f, 3.4f, 1.5f, 5.4f };
    public bool[] c = new bool[] { false, true, false, true };
}

public void Main()
{
    // A 类方式：引用数组连续，但实际对象数据分散在堆上
    A[] arrayA = new A[3] { new A(), new A(), new A() };

    Console.WriteLine("A class: a={0} b={1} c={2}", arrayA[0].a, arrayA[0].b, arrayA[0].c);
    
    // B 类方式：对象只持有一个索引，通过索引访问集中存储的数据
    B b = new B();
    b.index = 2;

    C c = C.instance;

    // 数据访问时，通过索引去值类型数组中读取，所有数据在内存中连续
    Console.WriteLine("B class: a={0} b={1} c={2}", c.a[b.index], c.b[b.index], c.c[b.index]);
}
```

上述代码中，`A` 类使用了我们非常熟悉的面向对象编程方式，把所有属性变量都放在自己身上，数据集合以引用的方式存储在数组中；而 `B` 类则将数据集中存放在了 `C` 类中，自己只存放一个索引值。

当两者都对数据进行存取时，`A` 类的数据在内存中是分散的——因为每次 `new A()` 时，都是从堆内存中找一块空地来分配，并不保证相邻实例的内存连续，`arrayA` 中只是引用连续，而非数据本身连续。而 `B` 类的数据则存储在值类型的数组中，值类型数组在分配时一定是内存连续的，这样就能更多地利用缓存，提高 CPU 读取数据时的命中率。

## 三、对象池

类实例在堆上分配内存后，当引用变量被置空或丢弃，这些内存就会交由**垃圾回收器（GC）**来回收。然而，内存分配本身是耗时的，垃圾回收的过程则更加昂贵。

简单来说，垃圾回收的过程是：当内存不够用时，GC 会遍历所有已分配的内存块，检查哪些类实例已经没有任何变量引用，然后将其回收。如果回收后依然内存不足，就会向操作系统申请更多内存。很明显，业务逻辑越庞大、数据量越多，GC 需要检查的内容就越多。

垃圾回收过程如此繁重，每次回收都会消耗大量 CPU 算力，因此我们不应该在 C# 中随意 `new` 创建类实例，然后又不管它们的死活随意丢弃或置空引用，让类实例不断地被创建和抛弃。

> **销毁实例 vs 置空引用：它们对 GC 的影响有何不同？**  
> 无论是将引用变量设为 `null`，还是让变量离开作用域被自然销毁，最终的效果是相同的——被引用的对象变得"不可达"。GC 会在下一次回收时标记这些不可达对象，并释放它们的内存。两者的区别在于时机：显式置空（`obj = null`）可以提前告诉 GC 该对象已不需要，让 GC 在下次回收时就能处理它；而离开作用域则是隐式的，在栈帧弹出后引用自然消失。对于**长生命周期**的对象（如静态字段、单例引用），如果某个被引用的对象不再需要了，应当显式地将引用置空——否则这个引用永远不会离开作用域，相关对象也会一直无法被回收。

### 小心 List 的使用

除了 `new` 某个类导致 GC 耗时增加外，以我的经验来看，很容易被我们忽略的是 `new List` 这种操作。我们在日常编程中大量使用动态数组，并且随时将它丢弃——因为它实在太容易创建和抛弃了。类似的，`Dictionary<int, List<T>>` 也是常被忽视的内存分配消耗来源之一：被装进 Dictionary 中的 List 经常被随意移除（Remove）并丢弃，却没有考虑过它是否能被再次利用。

### Unity 提供的通用对象池工具

我们应该尽可能使用**对象池**来重复利用已经创建的对象，而不是反复创建和销毁，这有助于减少内存分配的耗时，减少堆内存中的内存块数量，最终降低垃圾回收带来的 CPU 损耗。

下面两个对象池的类是从 Unity 的 UI 库中提取出来的，都是非常实用的对象池工具，我们应该尽可能使用它们。

```csharp
internal class ObjectPool<T> where T : new() // T 必须要有一个无参构造函数
{
    // 使用栈结构存储闲置对象，Push/Pop 操作均为 O(1)
    private readonly Stack<T> m_Stack = new Stack<T>();
    
    // 从池中取出对象时的回调，可用于初始化对象状态
    private readonly UnityAction<T> m_ActionOnGet;
    
    // 将对象归还池中时的回调，可用于重置对象状态
    private readonly UnityAction<T> m_ActionOnRelease;

    // 池总共创建过的对象数量
    public int countAll { get; private set; }
    // 当前正在使用的对象数量
    public int countActive { get { return countAll - countInactive; } }
    // 当前闲置在池中、可供取用的对象数量
    public int countInactive { get { return m_Stack.Count; } }

    public ObjectPool(UnityAction<T> actionOnGet, UnityAction<T> actionOnRelease)
    {
        m_ActionOnGet = actionOnGet;
        m_ActionOnRelease = actionOnRelease;
    }

    // 从池中获取一个对象（优先复用闲置对象，没有则新建）
    public T Get()
    {
        T element;
        if (m_Stack.Count == 0)
        {
            // 池中无闲置对象，创建新实例并增加总数计数
            element = new T();
            countAll++;
        }
        else
        {
            // 从栈顶弹出一个闲置对象复用
            element = m_Stack.Pop();
        }
        // 调用 Get 回调（如果有的话），例如重新初始化对象状态
        if (m_ActionOnGet != null)
            m_ActionOnGet(element);
        return element;
    }

    // 将对象归还到池中
    public void Release(T element)
    {
        // 简单的安全检查：防止连续重复归还同一个对象
        if (m_Stack.Count > 0 && ReferenceEquals(m_Stack.Peek(), element))
            Debug.LogError("Internal error. Trying to destroy object that is already released to pool.");
        // 调用 Release 回调（如果有的话），例如清空对象的状态
        if (m_ActionOnRelease != null)
            m_ActionOnRelease(element);
        // 将对象压入栈中以供后续复用
        m_Stack.Push(element);
    }
}

internal static class ListPool<T>
{
    // 为 List<T> 创建专用的对象池
    // Get 时不需要额外操作，Release 时自动清空列表内容
    private static readonly ObjectPool<List<T>> s_ListPool = new ObjectPool<List<T>>(null, l => l.Clear());

    public static List<T> Get()
    {
        return s_ListPool.Get();
    }

    public static void Release(List<T> toRelease)
    {
        s_ListPool.Release(toRelease);
    }
}
```

上述对象池使用栈（`Stack`）作为存储容器，将废弃的对象存储起来，在需要时从栈顶推出实例交给使用者。栈的出栈和入栈都是 O(1) 操作，非常高效。

对象池本身并不复杂，复杂的是使用——程序中所有需要创建对象实例、销毁对象实例、移除对象实例的地方，都需要用对象池来管理。我们来看几个使用上述 `ObjectPool` 和 `ListPool` 的例子：

```csharp
public class A
{
    public int a;
    public float b;
}

// 为 A 类型创建专用的对象池实例
internal static class PoolA
{
    private static readonly ObjectPool<A> s_Pool = new ObjectPool<A>(null, null);

    public static A Get() => s_Pool.Get();
    public static void Release(A obj) => s_Pool.Release(obj);
}

public void Main()
{
    // 示例 1：在 Dictionary 中管理对象池中的对象
    Dictionary<int, A> dic = new Dictionary<int, A>(16);
    for (int i = 0; i < 1000; i++)
    {
        // 从对象池中获取 A 实例，避免 new A() 导致的堆内存分配
        A a = PoolA.Get();
        a.a = i % 10;   // 键在 0-9 之间循环，因此会重复命中同一个键
        a.b = 3.5f;

        A item = null;
        if (dic.TryGetValue(a.a, out item))
        {
            // 键已存在（键会循环重复），覆盖前将旧对象归还对象池
            PoolA.Release(item);
        }

        dic[a.a] = a;

        // 随机移除某个键，被移除的对象归还对象池
        int removeKey = Random.RangeInt(0, 10);
        if (dic.TryGetValue(removeKey, out item))
        {
            PoolA.Release(item);
            dic.Remove(removeKey);
        }
    }

    // 示例 2：使用 ListPool 管理 List 的生命周期
    Dictionary<int, List<A>> dic2 = new Dictionary<int, List<A>>(1000);
    for (int i = 0; i < 1000; i++)
    {
        // 从 ListPool 中获取 List 实例，避免 new List<A>() 导致的堆内存分配
        List<A> arrayA = ListPool<A>.Get();

        dic2.Add(i, arrayA);

        List<A> item = null;
        int removeKey = Random.RangeInt(0, 1000);
        if (dic2.TryGetValue(removeKey, out item))
        {
            // 移除时将 List 归还对象池，Release 时 ListPool 会自动调用 Clear() 清空列表
            ListPool<A>.Release(item);
            dic2.Remove(removeKey);
        }
    }
}
```

上述代码中，`A` 类和 `List<A>` 各有 1000 次创建需求，每次创建都使用对象池，并在 Dictionary 移除时将对象送还对象池。这样一来，我们就能不断复用被"废弃"的对象，而不需要总是创建新的。所有被归还的对象都会被存储在池中，不会被垃圾回收器回收，最终内存被不断重复利用，从而减少了内存分配和释放带来的消耗。

## 四、字符串 String

字符串的性能问题在大部分语言中都是比较棘手的问题，C# 中尤其如此。在 C# 中 `string` 是引用类型，每次动态创建一个字符串时，C# 都会在堆内存中分配一块内存来存放字符串内容。我们来看看它到底有多"恐怖"。

```csharp
string strA = "test";
for (int i = 0; i < 100; i++)
{
    string strB = strA + i.ToString();        // 第 1 次内存分配：连接产生新字符串

    string[] strC = strB.Split('e');           // 第 2、3 次内存分配：分割产生两个新字符串

    strB = strB + strC[0];                     // 第 4 次内存分配：连接产生新字符串

    string strD = string.Format("Hello {0}, this is {1} and {2}.", strB, strC[0], strC[1]);
    // 第 5 次内存分配：Format 产生新字符串
}
```

这是一段"恐怖"的程序。循环中每一次迭代做了以下事情：

1. 将 `strA` 字符串和 `i` 的整数字符串连接，`strB` 获得的是从内存中**新分配**的字符串；
2. 将 `strB` 按 `'e'` 切割成 `strC`，切割出的两个字符串又**新分配**了两段内存；
3. 将 `strB` 与 `strC[0]` 连接，又再**申请**了一段内存来存放连接后的内容，并赋值给 `strB`，`strB` 原来指向的内容因为没有变量引用就被"丢弃"了；
4. 最后用 `string.Format` 将 4 个字符串组合起来，又**新分配**了一段内存来存放组合后的内容。

这里需要注意一点：**字符串常量不会被 GC 回收**。比如这段程序中的 `"test"` 和 `"Hello {0}, this is {1} and {2}."` 这两个常量，它们常驻于内存——即使没有变量指向它们，它们也不会被回收；下次使用时也不需要重新分配内存。

> **字符串常量与字符串驻留（String Interning）**  
> 在 C# 中，编译期的字符串字面量会被自动**驻留（Intern）**到 CLR 管理的字符串驻留池（Intern Pool）中。驻留池是一个全局的哈希表，存储了所有唯一的字符串字面量。当代码中出现相同的字符串字面量时，编译器会让所有引用指向驻留池中的同一块内存。这就是为什么 `Object.ReferenceEquals("hello", "hello")` 会返回 `true`——两个 `"hello"` 指向的是同一个驻留字符串。  
> 动态创建的字符串（如通过拼接、`StringBuilder` 等生成的字符串）**不会**被自动驻留。如果希望手动将动态字符串加入驻留池，可以调用 `string.Intern()` 方法。需要注意的是，驻留的字符串在应用程序域的整个生命周期内都不会被 GC 回收，因此不要滥用驻留功能，否则可能导致内存持续增长。

每次循环中，这段代码向内存申请了 5 次新的内存段，并且抛弃了 1 次（`strA + i.ToString()` 的结果因为没有变量指向而被丢弃）。这还不是最糟糕的——最糟糕的是，每次循环结束都会将前面分配的所有临时内存内容全部抛弃，重新再分配一次。如此反复申请和抛弃，总共向内存申请了约 500 次内存段，并全部抛弃，**内存被浪费得非常严重**。

为什么会这样呢？究其原因，C# 对字符串**没有任何缓存机制**，每次操作都需要重新分配 `string` 内容。据我所知，很多语言都没有字符串的缓存机制，因此我们的字符串连接、切割、组合等操作都会向内存申请新的内容，并抛弃没有变量指向的旧字符串，等待 GC 回收。我们知道 GC 一次回收会消耗大量 CPU，如果我们不注意字符串的使用方式、不断浪费内存，将导致程序不定时卡顿。并且随着程序运行时间越来越长、各模块中不良代码的积累，程序卡顿的频率会逐渐升高，运行效率也会越来越差。

### 如何解决字符串问题

#### 1. 自建缓存机制

可以用一些标志性的 Key 键值来一一对应字符串，比如游戏项目中常用 ID 来构造某个字符串：

```csharp
int ID = 101;

ResData resData = GetDataById(ID);

string strName = "This is " + resData.Name;

return strName;
```

> 一个 ID 对应一个字符串的场景，我们可以建立一个字典容器将其缓存起来，下次使用时就不需要重新申请内存了。

```csharp
Dictionary<int, string> strCache;

string strName = null;
if (!strCache.TryGetValue(id, out strName))
{
    // 缓存中不存在，创建并存入缓存
    ResData resData = GetDataById(id);
    strName = "This is " + resData.Name;
    strCache.Add(id, strName);
}

return strName;
```

我们用 `Dictionary` 容器将字符串缓存起来，每次先查询字典中是否存在，有则直接使用，没有则创建一个并将其存入字典，以便下次复用。

#### 2. 使用不安全代码直接操作内存

由于 `string` 类本身的任何修改操作都会申请新的内存，因此我们需要突破这个限制——直接使用指针来修改 `string` 中字符的值，这样就能重复利用已分配的 `string` 内存，不需要反复重新分配。

C# 虽然托管了大部分内存内容，但它也允许我们使用**非托管方式**来直接访问和修改内存——这就是 `unsafe` 关键字的用途。通过非托管方式，我们可以绕过 `string` 的不可变性，直接改变其中的字符内容，使已分配的字符串内存能被重复利用。来看一个例子：

```csharp
string strA = "aaa";

// 使用 "bbb" + "b" 的方式生成新字符串，是为了避免修改常量字符串内存块
string strB = "bbb" + "b";

// 使用 fixed 关键字固定指针所指向的对象的内存
fixed (char* strA_ptr = strA)
{
    fixed (char* strB_ptr = strB)
    {
        // 将 strA 的前 3 个字符内容直接拷贝到 strB 的内存中
        memcopy((byte*)strB_ptr, (byte*)strA_ptr, 3 * sizeof(char));
    }
}

Console.WriteLine(strB); // 此时 strB 内容变为 "aaab"
```

> 注意：这里特意用 `"bbb" + "b"` 而不是直接写 `"bbbb"`，是为了确保 `strB` 不是常量字符串内存块，而是运行时期动态分配的新字符串，这样我们修改它的内容才是安全的。

其中 `memcopy` 并非系统函数，需要自己实现，写法也很简单——拿到两个指针，根据长度遍历并逐字节赋值：

```csharp
public unsafe void memcopy(byte* dest, byte* src, int len)
{
    while ((--len) >= 0)
    {
        dest[len] = src[len];
    }
}
```

> **注意**：使用 `unsafe` 代码需要在项目设置中启用 "Allow unsafe code" 选项，且这类代码绕过了 C# 的内存安全检查，使用时需格外谨慎。

我们把 `strB` 的前 3 个字符的内容替换成了 `strA` 中的内容，整个过程并没有增加任何新内存，因为我们使用了不安全的非托管方法直接操控内存。通过这种方式，我们可以将已分配的 `string` 缓存起来重复利用。下面来看一个实际应用的例子：

```csharp
// 按字符串长度缓存可复用的字符串
Dictionary<int, string> cacheStr;

public unsafe string Concat(string strA, string strB)
{
    int a_length = strA.Length;
    int b_length = strB.Length;
    int sum_length = a_length + b_length;

    string strResult = null;

    if (!cacheStr.TryGetValue(sum_length, out strResult))
    {
        // 缓存中不存在该长度的字符串，直接连接后存入缓存
        strResult = strA + strB;
        cacheStr.Add(sum_length, strResult);
        return strResult;
    }

    // 缓存中存在可用字符串，用指针直接修改其内容
    fixed (char* strA_ptr = strA)
    {
        fixed (char* strB_ptr = strB)
        {
            fixed (char* strResult_ptr = strResult)
            {
                // 将 strA 的内容拷贝到 strResult 的开头
                memcopy((byte*)strResult_ptr, (byte*)strA_ptr, a_length * sizeof(char));

                // 将 strB 的内容拷贝到 strResult 中 a_length 偏移之后的位置
                memcopy((byte*)strResult_ptr + a_length * sizeof(char), (byte*)strB_ptr, b_length * sizeof(char));
            }
        }
    }

    return strResult;
}
```

当需要将两个字符串连接起来时，先在缓存中查找是否有可用长度的字符串：没有则直接连接并缓存；有则取出来，用指针方式直接修改缓存字符串的值。

#### 3. C# 自带方案

上述 `unsafe` 方案的核心思想是"预分配一块可复用的缓冲区，用的时候直接往里面写内容"，这是一种教学性质的极端优化演示。在实际开发中，C# 自身已经提供了更安全、更易用的替代方案，其内核与我们手动实现的方式完全一致。

##### StringBuilder 缓存

`StringBuilder` 内部维护一个 `char[]` 缓冲区，其工作原理与上述手动按长度缓存 `string` 的思路如出一辙：`Append` 往缓冲区末尾填字符，`Clear()` 只把内部写指针归零，**不释放底层的 `char[]` 数组**，下次使用时直接覆盖写入，零额外分配。

我们不需要为 `StringBuilder` 实现任何包装——只需要把 `StringBuilder` 实例本身缓存起来即可：

```csharp
// 缓存一个 StringBuilder 实例并反复使用
StringBuilder sb = new StringBuilder(256);  // 预估合理初始容量，避免内部数组扩容

sb.Append("Hello ");
sb.Append(name);
sb.Append(", this is ");
sb.Append(123);

string result = sb.ToString();

sb.Clear();  // 清空后下次直接复用，不产生新内存分配
```

> **注意**：`sb.ToString()` 仍然会产生一次新的 `string` 分配（这是不可避免的，因为你需要把结果拿出去）。但拼接过程中的中间字符串分配被完全消除了——这正是 `StringBuilder` 相比 `+` 拼接的优势所在。

在 Unity 项目中，可以结合对象池来做更全面的管理：

```csharp
internal static class StringBuilderPool
{
    // 使用前面介绍的 ObjectPool 来管理 StringBuilder 实例
    private static readonly ObjectPool<StringBuilder> s_Pool =
        new ObjectPool<StringBuilder>(null, sb => sb.Clear());

    public static StringBuilder Get() => s_Pool.Get();
    public static void Release(StringBuilder sb) => s_Pool.Release(sb);
}
```

每次需要拼接字符串时从池中取，用完后归还，避免反复创建 `StringBuilder` 实例。以下是一个实际使用的例子：

```csharp
// 模拟每帧生成日志字符串的场景
void BuildLogLine(string playerName, int score)
{
    StringBuilder sb = StringBuilderPool.Get();  // 从池中取出

    sb.Append("[");
    sb.Append(playerName);
    sb.Append("]  score: ");
    sb.Append(score);
    sb.Append(", rank: ");
    sb.Append(GetRank(score));

    string log = sb.ToString();                   // 转为 string 输出

    StringBuilderPool.Release(sb);               // 用完归还池中，Release 回调自动 Clear
    // 之后同一实例可被其他调用者复用，无需 new StringBuilder()
}
```

> 这个场景下，池中只需维持少数几个 `StringBuilder` 实例即可覆盖所有调用，对象复用率极高。

> `StringBuilder` 的初始容量建议根据实际场景预估。如果预估容量偏小，内部 `char[]` 扩容时会产生一次数组拷贝——这同样是内存分配，应尽量避免。如果不确定容量，宁可稍微偏大一些。

##### string.Intern 驻留池

对于**字符串种类有限但反复出现**的场景（比如配置表 key、网络协议字段名、资源路径等），可以利用 C# 自带的字符串驻留池（Intern Pool）：

```csharp
// 将动态字符串加入驻留池（如果池中已存在同内容的字符串，返回已有引用）
string cached = string.Intern(someDynamicString);

// 仅查找驻留池，不自动加入（不存在则返回 null）
string existing = string.IsInterned(someDynamicString);
```

驻留之后，所有相同内容的字符串都指向同一块内存，后续的 `==` 比较会命中 `ReferenceEquals` 而**直接返回 `true`**，彻底避免逐字符遍历——这正是后文"字符串比较的隐藏开销"一节中提到的优化方向。

> **注意**：驻留池的字符串在整个应用程序域生命周期内都不会被 GC 回收，因此只适合"种类有限"的场景。不要对高度动态变化、种类无穷的字符串（如用户输入、随机生成的内容）大量使用 `string.Intern`，否则驻留池会持续膨胀，反而导致内存泄漏。

| 方案 | 安全性 | 分配次数 | 适用场景 |
|------|--------|----------|----------|
| `StringBuilder` 缓存 | ✅ 安全 | 仅 `ToString()` 时 1 次 | 拼接、格式化等通用场景 |
| `string.Intern` | ✅ 安全 | 0（复用已有引用） | 种类有限的整串缓存（Key / 路径） |
| 手动 `unsafe` + 缓存 | ⚠️ 不安全 | 0 | 极端优化场景，不建议日常使用 |

### 字符串的其他隐藏问题

#### Length 属性并非 O(1)

`string` 的 `Length` 属性可不是省油的灯。由于 C# 的 `string` 类中只存放了 `char[]` 数据，没有单独存储长度值的字段，因此取长度时需要通过遍历字符串来找到结尾的 `\0` 字符。类似于如下伪代码：

```csharp
int sum = 0;
for (; data[sum] != '\0'; ++sum) ;
return sum;
```

> 如果我们大量使用 `Length`，就会浪费很多 CPU，因为每次调用它都会遍历整个字符串。好在实际开发中，我们通常不会在循环中反复访问同一字符串的 `Length`——一种简单的优化是将其缓存到局部变量中。

#### 字符串比较的隐藏开销

字符串比较也有隐藏的性能问题。当两个字符串进行比较时，`string` 的 `Equals` 方法首先会比较两个字符串的**引用指针**是否一致，一致则直接返回 `true`。如果指针不一致，则会先比较两者的**长度**是否相等，长度不同则直接返回 `false`；长度相同才会逐个遍历每一个字符，判断它们是否全部相等。我们来看看 `Equals` 的实际源码：

```csharp
public bool Equals(String value)
{
    if (this == null)                        // 防止 reverse-pinvoke 等场景的空引用
        throw new NullReferenceException();

    if (value == null)
        return false;

    if (Object.ReferenceEquals(this, value)) // 第一步：比较引用是否相同
        return true;
    
    if (this.Length != value.Length)         // 第二步：比较长度是否相同
        return false;

    return EqualsHelper(this, value);        // 第三步：逐字符遍历比较
}
```

> 这段代码从源码中提取。在判空之后，依次进行：引用比较 → 长度比较 → 逐字符遍历。只有当前两步都无法得出结论时，才会进入最昂贵的逐字符比较。

这意味着，如果两个字符串内容相同但来自不同的内存段（例如分别通过拼接得到），那么比较它们时就不得不遍历所有字符。如下示例：

```csharp
string strA = "Hello ";
string strB = "Hello ";

strA = strA + "C";   // strA 在堆上新分配一块内存
strB = strB + "C";   // strB 也在堆上新分配一块内存（与 strA 不同）

// 虽然 strA 和 strB 内容完全相同（都是 "Hello C"），但它们位于不同的内存段
// 引用比较（ReferenceEquals）会返回 false
// 因此 Equals 会进入逐字符遍历比较
if (strA == strB)
{
    Console.WriteLine("相等"); // 最终结果：相等，但经过了昂贵的遍历比较
}
```

> 在这个例子中，`strA` 和 `strB` 内容完全相同，但由于分别通过拼接产生，位于不同的内存地址，因此 `ReferenceEquals` 返回 `false`，`Equals` 不得不进入逐字符遍历。如果这样的比较频繁发生，累积的性能损耗也不容忽视。将字符串缓存起来（如前文所述），让相同内容的字符串指向同一块内存，就能让 `ReferenceEquals` 快速返回 `true`，从而避免遍历。
