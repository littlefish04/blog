    一、使用 List 和 Dictionary

    List 和 Dictionary 是 C# 中最常用的集合类型，它们的底层都是由数组实现的。理解它们的内部机制，才能避免在性能敏感的场景中踩坑。

    List

    List 在插入元素时，需要将插入位置之后的所有元素依次向后移动，因此 Insert 的时间复杂度是 O(n)。

    Contains 通过遍历数组来查找目标元素，时间复杂度也是 O(n)。

    Remove 同样需要先找到元素再移动后续元素，复杂度同样是 O(n)。

    因此，如果频繁调用 Insert、Contains、Remove 这些 O(n) 操作，尤其是在 List 容量较大的情况下，会带来很多不必要的性能消耗。在合适的场景下，可以考虑使用 HashSet、Dictionary 等 O(1) 查找的数据结构来替代。

    Dictionary

    在编写程序时应注意给 Dictionary 设置一个合理的初始容量，而不是任由它自行扩容。扩容时的数组重新分配不仅会加重 GC 的负担，过小的容量还会让哈希冲突变得更加频繁，进一步导致性能损耗。

    此外，如果使用类实例作为 Dictionary 的 Key，Dictionary 会使用 Object 的 GetHashCode() 方法来获取类实例的哈希值。而 GetHashCode() 是通过算法将内存地址转化为哈希值的，它并不会缓存计算结果——每次调用都会重新计算一次，这是一种比较隐蔽的性能损耗。如果频繁通过 GetHashCode 来提取 Value，我们应该关注一下 GetHashCode 的计算开销，考虑是否可以用唯一 ID（如 int 或 string）的方式来代替 GetHashCode 算法。

    二、使用 Struct

    Struct 是值类型。当作为局部变量使用时，Struct 分配的内存在栈上，生命周期结束后回收也十分简单——只需将栈的尾指针回退即可。与分配在堆上的类实例不同，值类型的分配和回收不会产生内存碎片，也不需要 GC 介入。

    另外，CPU 从栈上读取数据比从堆上读取数据更高效，因为栈内存具有更好的访问局部性。

    CPU 缓存命中和 Struct 数组

    CPU 在读取内存时，会把连续的一大块内容加载到缓存中。从缓存读取数据比直接从内存读取快得多（可达 100 倍以上）。下次读取数据时，CPU 会先在缓存中查找，如果命中则无需再次访问内存。

    连续内存在 CPU 读取时，缓存的命中率高；而非连续内存则命中率较低。我们应该尽可能让 CPU 读取数据时命中缓存，从而提升访问效率。

    正因为如此，Struct 数组对提高内存访问速度很有帮助。由于 Struct 是值类型，Struct 数组中的所有元素都是连续存储的——每个元素的数据实实在在”嵌”在数组的连续内存空间中。而 Class 数组则不同：虽然数组中的引用（指针）变量是连续的，但每个引用所指向的、位于堆上的实际数据则像碎片一样分散在各处。

    不过，并非所有 Struct 都能提高缓存命中率。如果一个 Struct 太大，超过了 CPU 一次缓存加载的数据块大小，那么缓存的效果就会大打折扣——因为一次加载的数据可能只包含一个甚至半个 Struct。因此，很多高性能架构彻底抛弃了在对象上直接存放数据的方式，转而使用值类型连续数组来提高 CPU 缓存命中率：将同一类数据集中到基本类型的数组中，而具体对象上只存放一个索引值，通过索引来间接访问数组中的数据。我们来看一个例子：

    三、对象池

    类实例在堆上分配内存后，当引用变量被置空或丢弃，这些内存就会交由**垃圾回收器（GC）**来回收。然而，内存分配本身是耗时的，垃圾回收的过程则更加昂贵。

    简单来说，垃圾回收的过程是：当内存不够用时，GC 会遍历所有已分配的内存块，检查哪些类实例已经没有任何变量引用，然后将其回收。如果回收后依然内存不足，就会向操作系统申请更多内存。很明显，业务逻辑越庞大、数据量越多，GC 需要检查的内容就越多。

    垃圾回收过程如此繁重，每次回收都会消耗大量 CPU 算力，因此我们不应该在 C# 中随意 new 创建类实例，然后又不管它们的死活随意丢弃或置空引用，让类实例不断地被创建和抛弃。

    销毁实例 vs 置空引用：它们对 GC 的影响有何不同？
    无论是将引用变量设为 null，还是让变量离开作用域被自然销毁，最终的效果是相同的——被引用的对象变得”不可达”。GC 会在下一次回收时标记这些不可达对象，并释放它们的内存。两者的区别在于时机：显式置空（obj = null）可以提前告诉 GC 该对象已不需要，让 GC 在下次回收时就能处理它；而离开作用域则是隐式的，在栈帧弹出后引用自然消失。对于长生命周期的对象（如静态字段、单例引用），如果某个被引用的对象不再需要了，应当显式地将引用置空——否则这个引用永远不会离开作用域，相关对象也会一直无法被回收。

    小心 List 的使用

    除了 new 某个类导致 GC 耗时增加外，以我的经验来看，很容易被我们忽略的是 new List 这种操作。我们在日常编程中大量使用动态数组，并且随时将它丢弃——因为它实在太容易创建和抛弃了。类似的，Dictionary<int, List<T>> 也是常被忽视的内存分配消耗来源之一：被装进 Dictionary 中的 List 经常被随意移除（Remove）并丢弃，却没有考虑过它是否能被再次利用。

    四、字符串 String

    字符串的性能问题在大部分语言中都是比较棘手的问题，C# 中尤其如此。在 C# 中 string 是引用类型，每次动态创建一个字符串时，C# 都会在堆内存中分配一块内存来存放字符串内容。我们来看看它到底有多”恐怖”。

    这是一段”恐怖”的程序。循环中每一次迭代做了以下事情：

    1. 将 strA 字符串和 i 的整数字符串连接，strB 获得的是从内存中新分配的字符串；
    2. 将 strB 按 'e' 切割成 strC，切割出的两个字符串又新分配了两段内存；
    3. 将 strB 与 strC[0] 连接，又再申请了一段内存来存放连接后的内容，并赋值给 strB，strB 原来指向的内容因为没有变量引用就被”丢弃”了；
    4. 最后用 string.Format 将 4 个字符串组合起来，又新分配了一段内存来存放组合后的内容。

    这里需要注意一点：字符串常量不会被 GC 回收。比如这段程序中的 "test" 和 "Hello {0}, this is {1} and {2}." 这两个常量，它们常驻于内存——即使没有变量指向它们，它们也不会被回收；下次使用时也不需要重新分配内存。

    字符串常量与字符串驻留（String Interning）
    在 C# 中，编译期的字符串字面量会被自动**驻留（Intern）**到 CLR 管理的字符串驻留池（Intern Pool）中。驻留池是一个全局的哈希表，存储了所有唯一的字符串字面量。当代码中出现相同的字符串字面量时，编译器会让所有引用指向驻留池中的同一块内存。这就是为什么 Object.ReferenceEquals("hello", "hello") 会返回 true——两个 "hello" 指向的是同一个驻留字符串。
    动态创建的字符串（如通过拼接、StringBuilder 等生成的字符串）不会被自动驻留。如果希望手动将动态字符串加入驻留池，可以调用 string.Intern() 方法。需要注意的是，驻留的字符串在应用程序域的整个生命周期内都不会被 GC 回收，因此不要滥用驻留功能，否则可能导致内存持续增长。

    每次循环中，这段代码向内存申请了 5 次新的内存段，并且抛弃了 1 次（strA + i.ToString() 的结果因为没有变量指向而被丢弃）。这还不是最糟糕的——最糟糕的是，每次循环结束都会将前面分配的所有临时内存内容全部抛弃，重新再分配一次。如此反复申请和抛弃，总共向内存申请了约 500 次内存段，并全部抛弃，内存被浪费得非常严重。

    为什么会这样呢？究其原因，C# 对字符串没有任何缓存机制，每次操作都需要重新分配 string 内容。据我所知，很多语言都没有字符串的缓存机制，因此我们的字符串连接、切割、组合等操作都会向内存申请新的内容，并抛弃没有变量指向的旧字符串，等待 GC 回收。我们知道 GC 一次回收会消耗大量 CPU，如果我们不注意字符串的使用方式、不断浪费内存，将导致程序不定时卡顿。并且随着程序运行时间越来越长、各模块中不良代码的积累，程序卡顿的频率会逐渐升高，运行效率也会越来越差。
